# Superconductor Mobile — v1.1 roadmap

Roadmap for the iOS companion after v1 (Mac bridge, pairing, Pi terminal, iOS 26 + glass UI).

---

## v1 baseline (shipped)

- Mac bridge (REST + WebSocket over `sc`)
- Pairing (manual host / port / token)
- Agents list (interactive filter; optional shell tabs)
- Pi terminal: snapshot + live stream + send + stop
- iOS 26 minimum, `AppIcon.icon`, Liquid Glass chrome

Design: [`2026-06-18-superconductor-mobile-design.md`](./2026-06-18-superconductor-mobile-design.md)

---

## Post-v1 reality check (2026-06-18)

**Shipped in this repo (initial commit):**
- Bridge (Bun): health, status, pairing (authed), agents list, snapshot, send/prefill/queue, interrupt, stop, basic worktree/status, agent WS stream (`sc agent subscribe --live-only`).
- iOS (SwiftUI + iOS 26 + Icon Composer glass): manual host/port/token/TLS pairing (Keychain), agents list + Pi filter + refresh, live terminal snapshot view (plain Text lines) + composer + stop. Uses glassEffect, no full ANSI.
- Protocol + sc-client wrappers.

**Not yet (even in v1 baseline mentions):**
- No QR code flow (print-pairing exists as JSON only).
- `/v1/worktree/status` only; no diff-summary, review-*.
- No chat sessions, layout views, groups, coordination-state.
- No auto-reconnect on WS drop.
- No first-class Mac surface for the bridge (still `bun run bridge:start`; no menu bar / status item).
- Pairing still requires copying token from `~/.superconductor-mobile/bridge.json` manually (chicken-egg for /pairing).

All work **must** go through `sc` JSON / subscribe only. No direct socket or SSH.

## v1.1 — recommended order

### 1. Mac Companion — "bridge runs inside the Mac app" (highest leverage)

**Context (third-party companion):** We do not control Superconductor.app. We are building an independent companion so people can use their existing Superconductor setup from their iPhone.

Goal: Turn the current "run a terminal command" bridge into a proper, always-available Mac experience that users install on the same machine as Superconductor.

Since the main app is native, we have a great path:

| Option | Description | Reality for us |
|--------|-------------|----------------|
| **First-party Swift menu-bar companion (recommended)** | Small menu-bar-only Swift app (status item, no Dock icon). It provides the HTTP/WS surface + beautiful QR pairing + status. Can launch or contain the adapter logic. | Fully in our control in this repo. Feels "inside the Mac" to the user. |
| **Improve + auto-launch the Bun bridge** | Keep Bun server, add proper LaunchAgent / login item + a tiny status UI. | Faster to ship initially, but less native feel. |
| **Wish-list for upstream** | One day Superconductor.app could expose the mobile surface itself. | Nice to have later; we can't rely on it. |

**Recommended path:** Build the **Swift menu-bar companion** (Option 1). The iOS side keeps talking the exact same contract. This directly delivers the spirit of "make the bridge run inside the mac app" for third-party users.

Key items now:

| Item | Details | Status |
|------|---------|--------|
| **Swift menu bar / status item companion** (we control this) | Native Swift (SwiftUI + AppKit). No dock icon (or accessory). Status item for "Superconductor Mobile". Shows QR, regenerate token, host info, health. Manages the network adapter so it is always available. | Not started |
| **Pairing QR + deep link** | Generate QR from the pairing payload in the menu bar. iOS side: camera permission + scanner (or paste). Support `superconductor-mobile://pair?host=...&port=...&token=...&tls=...`. Solve the current auth chicken-egg for initial payload (short-lived unauthed `/v1/pairing` or one-time code). | Partial (JSON only) |
| **Reliable hosting of the server** | Option 1: The Swift companion bundles or execs the existing Bun server logic. Option 2: Port the small server surface to Swift (using Foundation or NIO) over time. Either way the iOS contract is unchanged. | Current = manual Bun |
| **Tailscale + host discovery** | Use `tailscale ip -4` (if available) or best interface. Expose cleanly in pairing payload and UI. | Easy |
| **Optional TLS inside the companion** | Same as before. | Planned |

**Unlocks:** Bridge is just *there* whenever Superconductor is running. Pairing becomes a delightful one-scan experience from the Mac menu bar.

**Near-term concrete slice (Mac side, third-party):**
Detailed implementation plan: [`docs/plans/2026-06-18-v1.1-mac-menu-bar-companion.md`](./2026-06-18-v1.1-mac-menu-bar-companion.md)

- New macOS target (Swift menu bar / status item app). Call it the Superconductor Mobile Companion.
- Discovers the local Superconductor socket.
- Provides a status item with "Pair iPhone" → popover/window containing big QR code + instructions + "Copy token" + Tailscale/LAN host info.
- Manages the HTTP/WS server (either execs the existing Bun logic or will host a Swift implementation).
- Can be set to launch at login (or user launches it once).
- iOS pairing flow can immediately add QR scanner support.

This is fully deliverable by us without touching the main app.

### 2. Worktree & review (Superconductor’s other half)

Current bridge only has a stub:
```ts
// /v1/worktree/status?path=...
scJson(["worktree", "status", "--json"], { cwd: worktreePath })
```

| Item | `sc` backing | New bridge routes | iOS surface | Status |
|------|--------------|-------------------|-------------|--------|
| Worktree picker / status | `worktree status --json` (already used in SC) | `/v1/worktree/status?path=...` (enhance), maybe list open worktrees via `worktree status` in active context | New tab or sheet. Use target branch + changed files count. | Stub only |
| Diff summary screen | `worktree diff-summary --file PATH --json` | `/v1/worktree/diff-summary?path=...` | List of files + summary hunks. Read-only first. | Not started |
| Review checklist | `worktree review-checklist --json` | `/v1/worktree/review-checklist` | Checklist UI (done / todo items per file). | Not started |
| Comment list + reply (read-first) | `worktree review-list`, `review-get`, `review-set-status`, `review-add`, `review-reply` | Full review CRUD proxy routes | Threaded comments view, ability to reply + set status. Anchor to file:line. | Not started |

**iOS notes (mobile-design + apple-ui-designer):** Use native List + disclosure. Keep interactions simple (tap to expand). Secure storage already in place. One-handed friendly.

**Unlocks:** review from the couch without opening the Mac.

**Dep:** Needs reliable connection (see §4 reconnect). Consider worktree context from `sc worktree status` in current SC session.

### 3. Chat sessions (not only terminal Pi)

From `sc --help`:
- `sc chat list --json`
- `sc chat new`, `sc chat send SESSION_ID`, `sc chat events SESSION_ID`, `sc chat stop` etc.
- `sc layout run ... --ui chat`

| Item | `sc` backing | Bridge | iOS |
|------|--------------|--------|-----|
| List + create chats | `chat list`, `chat new --worktree ... --provider ... --ui chat` | `/v1/chats` (list + create) | Separate "Chats" list or unified with Agents (filter by ui). |
| Send + stream | `chat send`, `chat events` (or layout read/send) | `/v1/chats/:id/send`, WS `/v1/chats/:id/stream` | Message list (richer than terminal snapshot), composer. Reuse stream service pattern. |
| Stop / close | `chat stop`, `chat close` | Proxy | Toolbar actions. |

**Why separate from agents?** Agents today are mostly `id:stable...` terminal sessions. Chats + layout support non-terminal surfaces (`ui: "chat"`).

**Unlocks:** agents that aren’t terminal-backed.

**Note:** `sc layout ...` commands also drive chat/terminal panes in workspaces. Consider exposing `layout views --json` early.

### 4. Session UX

Already have basic refreshable on Agents + initial snapshot + live WS.

| Item | Why / How | Notes |
|------|-----------|-------|
| **Reconnect** when WebSocket drops (exponential backoff) | `AgentStreamService` currently sets `isConnected=false` on error and stops. Add retry loop (1s, 2s, 5s, max 30s). Re-apply snapshot on reconnect. | Highest UX win. Use `Task` + cancellation. |
| **Pull-to-refresh** on terminal (snapshot) | Already on list. Add manual "Refresh snapshot" button in session that calls BridgeAPI.snapshot and applies. | Good for when live lags. |
| **Haptic** on send / error / state change | `UINotificationFeedbackGenerator` or `UIImpactFeedbackGenerator`. | Small native touch. Keep subtle. |
| **Connection health** on Agents (bridge + SC status) | Poll `/v1/status` + bridge health periodically. Show banner or per-agent dot. | Re-use existing status call. Consider `coordination-state` for shared "Mac awake" signal. |
| Background / lifecycle | Handle app background/foreground to pause/resume streams. | iOS best practice. |

**Already present:** `.refreshable`, live dot (green/orange), error alerts.

**Unlocks:** reliable daily use away from desk.

**Mobile constraint:** respect battery/network. Don't poll too aggressively when backgrounded. Use reachability if available.

## Suggested v1.1 MVP slice

Smallest set that feels like a real product upgrade (pick **one track** to ship first):

1. **Daily driver foundations**
   - Swift menu bar companion app (status item) that hosts the bridge + shows QR
   - QR code generation + iOS scanner + deep link support
   - Reconnect + connection banner/health in AgentSessionView + AgentsView
   - Auto-start / login item behavior so users don't think about it

2. **Review from couch (high leverage for Superconductor users)**
   - Bridge: proxy `worktree status`, `diff-summary`, `review-checklist`, `review-list`
   - iOS: new Worktree tab or modal. Start read-only status + diff summary list. Add checklist.
   - Use existing worktreePath from `sc` context (pass via status or new `/v1/worktree/current`)

3. **Polish**
   - Haptics, better error states, background handling, pull-to-refresh snapshot in session.
   - Connection health using periodic `/v1/status`.

**Verification for MVP:**
- Pair with QR on real device + Tailscale.
- See live Pi, send, watch reconnect after sleep.
- See at least worktree changed files count from phone.

## Principles (unchanged)

1. Official `sc` / local API only — no SSH, tmux, or socket hacks
2. Native SwiftUI, iOS 26+, honest terminal snapshots (no fake PTY)
3. Bridge token treated like a password; prefer Tailscale / private network
4. Follow mobile-design: touch targets, battery aware, Keychain already used, no desktop assumptions.
5. Apple HIG + apple-ui-designer: calm, system typography, glass only where it exists in iOS 26, clarity over decoration.
